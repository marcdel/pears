🍐🎉 Whimsy Mode Celebrations

Design spec · 2026-06-10 · Pears

Whimsy Mode shipped as an April Fools toggle that does nothing. This design gives it a job: when enabled for a team, three celebration effects activate on the pairing board. When disabled, the app behaves byte-for-byte identically to today.

Guiding principle All effects are gated on the existing :whimsy_mode FunWithFlags flag (per-team actor, already persisted and toggled by the UI switch). The toggle remains the single source of truth; no new state is introduced.

Shared Infrastructure

  • New dependency: canvas-confetti (~10 kb, zero transitive deps), vendored as assets/vendor/canvas-confetti.js (pinned ESM build) — this repo has no npm toolchain; JS deps live in assets/vendor/ alongside topbar and Sortable. Handles confetti bursts and star/sparkle particle shapes.
  • New module: assets/js/whimsy.js exporting three functions — confettiBurst(), sparklePoof(x, y), and drumroll(pearIds). All celebration code lives here, isolated from board logic.
  • Server → client signaling: LiveView push_event/3 with window.addEventListener("phx:whimsy:...") listeners registered in app.js. The server only pushes when whimsy is on, so server-triggered effects need no client-side flag check.
  • Client-side gate: the board root element renders data-whimsy={@whimsy_mode} from the existing assign, for the one effect (drop sparkles) that is client-initiated.

The Three Celebrations

1. Confetti on Save server-gated

Saving the day's pairs is the ritual's "done" moment — it deserves fanfare. In the record-pears handler, on success and when whimsy is enabled: 

socket = push_event(socket, "whimsy:confetti", %{})

The client listener fires a center-screen confetti burst. Roughly a two-line server change plus one listener.

2. Sparkles on Drop client-gated

When a pear lands in a track, a small ✨ poof appears at the drop point, using canvas-confetti's star particles.

The board has two drag systems: the legacy Pear/Destination hooks and the Sortable-based new_drag_n_drop path. Both drop handlers call sparklePoof(x, y), gated by the data-whimsy attribute (drops are client-initiated, so the server can't push this one).

3. Suggest Drumroll — Staggered Fly-In server-triggered, CSS-animated

In the recommend-pears handler, when whimsy is enabled, collect the ids of pears that were newly assigned (ids, not names — pear card DOM ids are built from pear.id) and push them in board order: 

socket = push_event(socket, "whimsy:drumroll", %{pears: assigned_pear_ids})

The client listener finds each pear card by DOM id and applies a whimsy-pop class with animation-delay: index × 150ms. One scale-bounce keyframe animation is added to app.css. The class is removed on animationend so repeat suggestions re-animate cleanly.

No fake latency LiveView renders the new assignments immediately; the cards simply arrive theatrically. Nothing blocks or delays the actual state change.

Error Handling

Celebrations are fire-and-forget visuals. If canvas-confetti fails to load or a listener throws, the effect silently no-ops and the board functions normally. No celebration code sits on any critical path (save, drag-drop, or suggest logic runs first; effects trail).

Testing

LayerApproach
LiveView (Elixir) assert_push_event that whimsy:confetti and whimsy:drumroll fire when the flag is on; refute_push_event when off. Assert the data-whimsy attribute renders true/false with the flag.
JS (whimsy.js) Deliberately thin, fire-and-forget wrapper around canvas-confetti; verified manually in the browser.
